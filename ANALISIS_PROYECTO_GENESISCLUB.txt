ANALISIS TECNICO DEL PROYECTO GENESISCLUB
=========================================

Fecha del analisis: 2026-04-10
Base del documento: lectura del codigo actual del repositorio


1. DESCRIPCION GENERAL DEL SISTEMA
----------------------------------

GenesisClub es una plataforma web para gestionar una comunidad o club con tres perfiles principales:
- Administrador
- Socio
- Jugador

Ademas existe un flujo publico para invitados o postulantes que todavia no tienen cuenta.

La idea general del sistema es:
- Recibir solicitudes publicas para convertirse en socio o jugador.
- Permitir que un administrador revise esas solicitudes y las apruebe o rechace.
- Crear usuarios reales solo cuando una solicitud es aprobada.
- Gestionar rubros o actividades del club.
- Permitir que los socios se relacionen con rubros, hagan solicitudes de acceso y envien invitaciones.
- Mantener una red de referidos entre socios, donde un socio puede invitar a otro.

En otras palabras, el sistema combina:
- Gestion de identidad y roles.
- Workflow de aprobacion de solicitudes.
- Gestion de rubros.
- Invitaciones por email.
- Seguimiento de relaciones entre socios.


2. ARQUITECTURA DEL PROYECTO
----------------------------

El repositorio esta organizado como un monorepo con dos aplicaciones separadas:

- `genesisClub-back-end`
  Backend REST hecho con Spring Boot.

- `genesisclub-fond-end`
  Frontend SPA hecho con Angular.

Arquitectura de alto nivel:

Cliente Angular
-> Servicios HTTP
-> API REST Spring Boot
-> Servicios de negocio
-> Repositorios JPA
-> Base de datos MySQL

Arquitectura backend:

- Capa Controller
  Expone endpoints REST para autenticacion, solicitudes, rubros, socios, jugadores, invitaciones y auditoria.

- Capa Service
  Contiene la logica de negocio real. Aca se validan solicitudes, se generan JWT, se crean usuarios y se relacionan entidades.

- Capa Repository
  Usa Spring Data JPA para consultar y persistir datos.

- Capa Modelo
  Se divide en entidades JPA, DTOs y enums.

- Capa de seguridad
  Usa Spring Security con JWT stateless y un filtro propio (`JWTAuthorizationFilter`).

Arquitectura frontend:

- Angular standalone components
  No hay modulos Angular clasicos; la app usa componentes standalone y rutas.

- Rutas por perfil
  El frontend esta separado conceptualmente en publico, administrador, socio y jugador.

- Servicios compartidos
  Encapsulan las llamadas HTTP, autenticacion, guardas y notificaciones.

- Componentes compartidos
  Hay modales y notificaciones reutilizables.

Punto tecnico importante:

El frontend usa `provideZonelessChangeDetection()`. Eso explica por que varios componentes llaman manualmente a `detectChanges()` o usan `signal()`. No es una arquitectura incorrecta, pero si agrega complejidad operativa.

4. EXPLICACION DE CADA MODULO O COMPONENTE IMPORTANTE
-----------------------------------------------------

4.1 Frontend publico

- `inicio`
  Landing page inicial.

- `login`
  Formulario de autenticacion. Llama a `AuthServicio`, guarda el token en `localStorage` y redirige segun el rol.

- `solicitud-socio`
  Formulario publico para postularse como socio. Envia los datos al backend y opcionalmente registra vehiculo.

- `solicitud-jugador`
  Flujo equivalente para postularse como jugador.

- `registro-invitado`
  Permite completar una solicitud usando un token de invitacion recibido por email.

4.2 Frontend administrador

- `inicio-admin` y `header-admin`
  Shell basico de navegacion del perfil administrador.

- `solicitudes-pendientes`
  Lista solicitudes pendientes de socios y permite aprobar o rechazar con modal de confirmacion.

- `solicitudes-jugadores-pendientes`
  Misma idea para jugadores.

- `solicitudes-rubros-pendientes`
  Gestiona solicitudes de acceso a rubros.

- `lista-socio` y `ver-socio`
  Permiten listar socios, ver detalle, consultar vehiculos, asociar o desasociar rubros y cambiar estado del socio.

- `lista-jugador` y `ver-jugador`
  Permiten revisar jugadores y cambiar su estado.

- `listar-rubros`, `crear-rubros`, `ver-rubro`
  CRUD funcional de rubros.

- `ver-res-por-socio`
  Muestra la red descendente de socios invitados por un socio dado.

4.3 Frontend socio

- `inicio-socio`
  Pantalla inicial del socio.

- `herader-socio`
  Encabezado del socio. Incluye logout y el modal para enviar invitaciones por email.

- `rubros-disponibles`
  Lista rubros activos y permite iniciar una solicitud de acceso mediante clave.

- `solicitar-rubro`
  Flujo puntual para enviar la solicitud de acceso a un rubro.

- `rubros-socio`
  Mezcla dos conceptos:
  consulta de rubros asociados al socio y asociacion por clave de acceso.

- `entrar-rubro-socio`
  Pantalla de detalle basica del rubro asociado.

Componentes de socio incompletos o poco desarrollados:
- `mis-socios`
- `invitaciones-socio`

4.4 Frontend jugador

- `inicio-jugador`
  Pantalla inicial del jugador.

- `rubros-jugador`
  Lista rubros y deja ingresar una clave para entrar.

- `entrar-rubro-jugador`
  Carga el detalle de un rubro por ID.

Este flujo existe visualmente, pero su integracion con seguridad y backend no esta del todo alineada.

4.5 Servicios compartidos frontend

- `AuthServicio`
  Maneja login, logout, lectura de token, decodificacion del rol e ID y redireccion por perfil.

- `AuthGuard`
  Protege rutas en base a login y rol.

- `auth.interceptor`
  Adjunta el header `Authorization: Bearer ...` a las llamadas HTTP cuando existe token.

- `SolicitudServicio`
  Encapsula el alta de solicitudes de socio, jugador e invitado.

- `RubroServicio`, `RubroSocioServicio`, `UsuarioRubroServicio`, `HistorialRubroServicio`, `RubroAccesoLogServicio`
  Encapsulan el acceso a los modulos de rubros y auditoria.

- `NotificacionesServicio`
  Implementa toasts globales para feedback al usuario.

4.6 Componentes compartidos frontend

- `modal-confirmacion`
  Modal reutilizable para confirmar aprobar, rechazar o ejecutar acciones riesgosas.

- `modal-solicitud-rubro`
  Modal para ingresar la clave de acceso de un rubro.

- `notificaciones`
  Renderiza el stream global de toasts.

4.7 Backend de autenticacion y seguridad

- `LoginController`
  Expone login, logout y un endpoint de prueba solo admin.

- `AuthServiceImpl`
  Verifica credenciales, calcula el perfil real del usuario y genera el JWT.

- `JWTUtilityServiceImpl`
  Firma y valida tokens RS256. El token incluye `rol` y tambien `socioId`, `jugadorId` o `adminId`.

- `SecurityConfig`
  Define seguridad stateless, CORS, rutas publicas y rutas por rol.

- `JWTAuthorizationFilter`
  Lee el token del header, lo valida y carga la autenticacion en Spring Security.

4.8 Backend de usuarios, solicitudes y altas

- `SolicitudSocioController` + `SolicitudSerSocioServiceImpl`
  Reciben la postulación a socio, validan email, vehiculo y patente, y guardan una `SolicitudEntity`.
  Si el admin aprueba, se crea el usuario real y, si corresponde, se genera la relacion con el socio padre.

- `SolicitudJugadorController` + `SolicitudJugadorServiceImpl`
  Mismo concepto para jugadores.

- `RegistroController` + `RegistroUsuarioServicioImpl`
  Implementan el registro de usuarios y la creacion de su entidad por rol:
  `AdminEntity`, `SocioEntity` o `JugadorEntity`.

4.9 Backend de invitaciones y red de socios

- `InvitacionController` + `InvitacionServiceImpl`
  Crean invitaciones con token UUID, vencimiento y envio de correo.

- `RelacionSocioController` + `RelacionSocioServiceImpl`
  Reconstruyen la red de referidos, tanto como arbol como lista plana.

4.10 Backend de rubros

- `RubroController` + `RubroServiceImpl`
  CRUD principal de rubros.

- `SolicitudRubroController` + `SolicitudRubroServiceImpl`
  Manejan la solicitud de acceso de un socio a un rubro. Al aprobar una solicitud se crea la relacion `RubroSocio`.

- `RubroSocioController` + `RubroSocioServiceImpl`
  Gestionan la relacion directa entre socio y rubro.

- `UsuarioRubroController`
  Existe como modulo CRUD adicional, pero no parece estar integrado de forma consistente con el flujo principal actual.

- `HistorialRubroController` y `RubroAccesoLogController`
  Son modulos de auditoria y registro de actividad.

4.11 Modelo de datos principal

Entidades mas importantes:
- `UsuarioEntity`: usuario base con email, password, telefono, rol y vehiculos.
- `SocioEntity`: extension del usuario para socios.
- `JugadorEntity`: extension del usuario para jugadores.
- `SolicitudEntity`: postulación pendiente para socio o jugador.
- `InvitacionEntity`: invitacion emitida por un socio.
- `RelacionUsuarioEntity`: relacion sponsor -> referido.
- `RubroEntity`: actividad o categoria del club.
- `RubroSocioEntity`: vinculo aprobado entre socio y rubro.
- `SolicitudRubroEntity`: pedido de acceso a rubro.
- `VehiculoEntity`: datos del vehiculo asociado al usuario o a la solicitud.
- `HistorialRubroEntity` y `RubroAccesoLogEntity`: auditoria.

4.12 Modulos presentes pero sin desarrollo real

En backend existen controladores base o placeholders para:
- `AdminController`
- `PerfilController`
- `RolController`
- `UbicacionController`
- `UsuarioController`
- `VehiculoController`
- `NotificacionController`

Esto indica que el sistema tiene una base de crecimiento, pero no todo lo que aparece en la estructura esta implementado.


5. FLUJO DE FUNCIONAMIENTO
--------------------------

5.1 Flujo de registro publico de socio

1. El visitante completa el formulario Angular.
2. `SolicitudServicio` envia la data a `/api/solicitud/socio/nuevo`.
3. `SolicitudSerSocioServiceImpl` valida email, patente y vehiculo.
4. Se guarda una `SolicitudEntity` con estado `PENDIENTE`.
5. El administrador abre `solicitudes-pendientes`.
6. Si aprueba, el backend crea `UsuarioEntity` y `SocioEntity`.
7. Si la solicitud venia de una invitacion, tambien se genera la relacion con el socio padre.

5.2 Flujo de registro publico de jugador

1. El visitante completa el formulario de jugador.
2. Se crea una `SolicitudEntity` con tipo `JUGADOR`.
3. El administrador la revisa desde la bandeja de pendientes.
4. Si aprueba, el backend crea `UsuarioEntity` y `JugadorEntity`.

5.3 Flujo de login

1. El usuario ingresa email y password.
2. Angular llama a `/api/auth/login`.
3. `AuthServiceImpl` verifica credenciales.
4. Se genera un JWT con rol y IDs del perfil.
5. El frontend guarda el token en `localStorage`.
6. El interceptor lo adjunta en cada request futura.
7. Las guardas de Angular y Spring Security limitan el acceso por rol.

5.4 Flujo de invitacion

1. Un socio envia una invitacion desde el header.
2. El backend genera un token UUID y envia un email usando Thymeleaf.
3. El invitado abre la URL del frontend.
4. Completa el registro de invitado.
5. El backend transforma eso en una solicitud de socio asociada a la invitacion.
6. El admin aprueba la solicitud.
7. Se crea el nuevo socio y se registra la relacion sponsor -> invitado.

5.5 Flujo de rubros para administrador

1. El admin crea o edita rubros.
2. Puede activarlos o desactivarlos.
3. Puede ver solicitudes de acceso a rubros.
4. Al aprobar una solicitud, el socio queda vinculado al rubro mediante `RubroSocioEntity`.

5.6 Flujo de rubros para socio

1. El socio consulta rubros activos.
2. Puede intentar asociarse por clave o enviar una solicitud de acceso.
3. El backend valida la clave y el estado del rubro.
4. Si la solicitud es aprobada, aparece la relacion socio-rubro.
5. Desde la pantalla de rubros asociados puede entrar al rubro.

5.7 Flujo de datos resumido

El patron general del sistema es:

Angular component
-> Angular service
-> HttpClient
-> Controller Spring
-> Service Spring
-> Repository JPA
-> Base de datos
-> DTO o entidad
-> respuesta HTTP
-> actualizacion visual del componente


6. POSIBLES MEJORAS O PUNTOS DEBILES
------------------------------------

1. Hay problemas de seguridad importantes alrededor de los rubros.
   `RubroDTO` expone `claveAcceso` al frontend y el endpoint `/api/rubro/activos` es publico. Ademas, en `rubros-jugador` la clave se valida del lado cliente. Eso hace que la clave de acceso deje de ser realmente secreta.

2. La clave privada JWT esta versionada dentro del proyecto.
   El archivo `src/main/resources/jwtKeys/private_key.pem` no deberia vivir en el repositorio ni empaquetarse asi para produccion.

3. Se registran datos sensibles en logs.
   Hay `System.out.println` de passwords en backend y `console.log` de password o token en frontend. Eso es riesgoso.

4. El logout no invalida realmente el token.
   Existe un mapa de tokens revocados en `AuthServiceImpl`, pero el filtro JWT nunca consulta esa lista. En la practica el token sigue siendo valido hasta expirar.

5. Hay rutas frontend sin proteccion consistente.
   Algunas vistas de socio o de entrada a rubros no tienen `canActivate`, aunque dependen de recursos protegidos. Eso puede generar accesos rotos o UX confusa.

6. El flujo de jugador en rubros parece incompleto o inconsistente.
   El jugador entra a un detalle que consulta `/api/rubro/{id}`, pero en seguridad ese GET no esta habilitado para `ROLE_JUGADOR`.

7. El modulo `UsuarioRubro` parece legado o parcialmente integrado.
   Convive con `RubroSocio`, pero el flujo principal de aprobacion usa `RubroSocio`. Eso sugiere modelo duplicado o en transicion.

8. Hay modulos y pantallas a medio implementar.
   Ejemplos claros: `Invitaciones` admin, `InvitacionesSocio`, `MisSocios` y varios controladores placeholder del backend.

9. Falta una estrategia de errores mas uniforme.
   En algunas pantallas se usan toasts y modales; en otras se siguen usando `alert()`, `confirm()` y `console.log()`.
   En backend, muchas validaciones lanzan `RuntimeException` sin un manejador global estandarizado.

10. La cobertura automatizada es baja.
    En backend solo existe un `contextLoads`.
    En frontend la mayoria de los specs son pruebas generadas tipo `should create`.

11. Hay deuda de performance evitable.
    Por ejemplo, en `SolicitudRubroServiceImpl` se hacen validaciones con `findAll().stream()` cuando ya existen repositorios capaces de consultar de forma mas precisa.

12. Hay inconsistencia en el modelo de estado del usuario.
    Los socios usan una entidad `EstadoSocioEntity`, mientras que los jugadores cambian un string simple en `UsuarioEntity.estado`.

13. La invitacion no esta validada de forma completamente robusta en el registro.
    El endpoint de aceptacion existe, pero el frontend no lo usa. Ademas, el alta desde invitacion no valida con la misma fuerza el estado consumido del token ni el email destinatario.

14. La configuracion podria estar mas desacoplada.
    `JWTUtilityServiceImpl` lee rutas hardcodeadas desde classpath en lugar de usar de verdad las propiedades `jwt.private-key-path` y `jwt.public-key-path`.


7. CONCLUSION Y RESUMEN SIMPLIFICADO
------------------------------------

GenesisClub es una aplicacion full stack con una base bastante clara:
- Angular para la experiencia de usuario.
- Spring Boot para la API y la logica de negocio.
- MySQL como persistencia.
- JWT para autenticacion.

Lo mas importante del negocio hoy es:
- recibir solicitudes,
- aprobar usuarios,
- invitar socios,
- administrar rubros,
- y vincular socios con rubros y con otros socios.

El proyecto se entiende rapido si uno lo mira asi:

- Publico:
  formularios de ingreso al sistema.

- Administrador:
  aprueba, lista, activa, bloquea y gestiona rubros.

- Socio:
  invita gente y se relaciona con rubros.

- Jugador:
  tiene un flujo mas simple, hoy menos maduro que el de socio.

Como estado general, el sistema ya tiene una estructura real de producto y no de demo:
- hay seguridad,
- hay modelo de dominio,
- hay workflows,
- hay integracion de email,
- y hay separacion entre capas.

Pero todavia arrastra varias deudas importantes:
- seguridad de claves de rubro,
- manejo de secretos,
- modulos incompletos,
- duplicacion de conceptos,
- y poca cobertura de tests.

Resumen ejecutivo en una frase:

GenesisClub ya funciona como una plataforma administrativa y operativa para un club con roles y aprobaciones, pero todavia necesita una segunda pasada fuerte de endurecimiento tecnico para quedar solido en seguridad, consistencia funcional y mantenibilidad.
